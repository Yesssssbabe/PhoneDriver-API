PhoneDriver-API 安全审计修复报告
=====================================

1. 项目 / 审计背景
-------------------
- 项目：PhoneDriver-API（基于云端视觉大模型的 Android 手机自动化框架）
- 审计报告：PhoneDriverAPI_40Agent_AuditReport.txt（2026-06-30）
- 修复范围：本次修复以审计报告中的 Critical（致命）与 High（高危）漏洞为主，对可安全落地、不影响现有测试集的漏洞进行了代码级修复；对需要架构改造或依赖外部设施的漏洞，列入“未修复”并给出建议。
- 约束：未修改 wechat_qr.jpg、未修改任何 README/Markdown 文件、未执行任何 git 操作、未触碰项目根目录之外的文件。

2. 已修复漏洞（按 ID / 严重程度 / 简述）
----------------------------------------
[0003] Critical  - ConfigResolver 未对 max_cycles 设置上限 → 已在 phone_agent.py 中限制为 [1, 50]。
[0004] Critical  - max_retries 无上界且 429 不计入熔断器 → 已限制 [0, 10]，且最终 429 会计入熔断失败。
[0006] Critical  - phone_agent.log 存在符号链接攻击 → _setup_logging 与 SecureRotatingFileHandler 现在会拒绝符号链接，并通过 umask 0o077 创建文件。
[0007] Critical  - API Key 长期驻留 os.environ（/proc/<pid>/environ 泄露） → ConfigResolver 与 ui.py 读取后即刻 del/unsetenv 删除。
[0010] Critical  - 缺少模型训练 opt-out 控制（部分修复） → OpenAICompatibleProvider 增加 training_opt_out 标志位，OpenRouter 头已做清洗；OpenAI 的 store:false 等 provider 专属字段需后续按接口文档补充。
[0011] Critical  - 图片缓存时间侧信道 → 默认禁用图片缓存（_max_cache_size=0）。
[0012] Critical  - 图片缓存导致敏感截图残留在堆内存 → 同上，缓存已禁用，且 _encode_image 读取后做最佳努力覆写。
[0013] Critical  - 任务描述存在提示注入 → 增加 _sanitize_user_request，移除控制字符与常见注入模式，并用 <task> 定界符包裹。
[0014] Critical  - 系统提示缺少视觉/文本注入抵抗 → DEFAULT_SYSTEM_PROMPT 增加 SECURITY RULES，要求模型忽略截图内指令。

[0016] Critical  - HTTP session close 与请求并发竞态 → OpenAICompatibleClient 增加 _session_lock + _closing 标志，关闭后拒绝新请求。
[0017] Critical  - Gradio API 端点即使 show_api=False 仍暴露 → launch 增加 api_open=False，并在 build_ui 中限制 queue(max_size=10)。

[0023] Critical  - 截图多线程竞态 → ScreenshotCapture 增加 RLock，capture_with_resize 整体加锁。
[0024] Critical  - resize 时零尺寸图导致除零 → 已增加 original_max == 0 保护。
[0025] Critical  - os.replace 覆盖硬链接目标 → 临时文件改用 tempfile.mkstemp(O_EXCL)，replace 前检查目标 st_nlink。
[0026] Critical  - adb pull 通过硬链写越权文件 → capture 用 O_CREAT|O_EXCL 预创建文件并校验 nlink==1；_encode_image 用 O_NOFOLLOW 打开。

3. 未修复漏洞（需要总工程师决策/架构改造）
------------------------------------------
[0001] Critical - 缺少精确的成本/Token 预算控制
  原因：当前 HTTP 客户端仅返回文本内容，未解析 usage 与模型单价表；粗略的 max_api_calls 计数已加，但真正的 USD 预算需改造 HttpClient 返回 usage 并维护成本表。
  请总工程师决策：是否引入 usage 解析 + 每模型单价配置，或接入 provider billing API。

[0002] Critical - 单周期操作无硬超时
  原因：仅靠周期内截止时间检查无法中断已阻塞的 ADB/HTTP 调用；完整修复需为 ADB 与 HTTP 请求引入可取消的并发（threading + future.cancel 或信号量）。
  请总工程师决策：是否接受“软超时”方案，或要求实现可强制中断的线程模型。

[0005] Critical - 截图前未检测敏感内容（银行/短信/OTP）
  原因：需要 dumpsys window / OCR 检测 + 误判控制，可能显著影响任务可用性。
  请总工程师决策：是否增加敏感内容检测层并允许用户配置白名单应用。

[0008] Critical - 未对 AI 响应做完整性校验（HMAC/签名）
  原因：需要 provider 侧配合签名，或本地引入受信任的签名代理。
  请总工程师决策：是否与各云厂商协商响应签名方案，或引入本地 MITM 代理做校验。

[0009] Critical - 缺少数据保留协议（DPA）与全 provider 训练 opt-out
  原因：各家 provider 的 opt-out 机制不同（OpenAI store、OpenRouter header、Kimi/Moonshot 政策不明），部分需用户账户层面配置。
  请总工程师决策：是否维护 provider 合规矩阵并在 UI 中强制用户确认。

[0018] Critical - ADB 未做应用级设备授权
  原因：需要引入用户-设备配对注册表及配对流程，改动较大。
  请总工程师决策：是否增加 device authorization registry 和配对 UI。

[0019] Critical - 未拒绝 ADB over TCP
  原因：直接拒绝 TCP 会阻断部分合法 Wi-Fi 调试场景；应做成可配置策略。
  请总工程师决策：是否默认仅允许 USB，并通过 PHONEDRIVER_ALLOW_TCP=1 显式放行。

[0032] High    - 连续录屏缺少每次同意机制
  原因：若强制每次同意将破坏 CLI/自动化体验，需要设计 UI  consent 弹窗或录制指示器。
  请总工程师决策：是否在 UI 增加显式同意复选框与视觉录制指示。

[0033] High    - OTP/SMS 通知未打码
  原因：需要 OCR 或通知栏检测，成本高且可能误伤正常任务。
  请总工程师决策：是否引入 OCR 敏感区域检测并允许配置屏蔽区域。

[0034] High    - 单 provider 无故障转移/负载均衡
  原因：需要 ProviderPool/FailoverProvider 架构及多 key 配置。
  请总工程师决策：是否实现多 provider 路由与优先级配置。

[0035] High    - API Key 缺少作用域/轮换/细粒度配额
  原因：依赖 provider 侧 key 管理（project key、scoped key），应用层只能做基础校验。
  请总工程师决策：是否增加 key 作用域校验与轮换模块。

[0040] High    - 任务循环各退出路径时间侧信道
  原因：强制统一时间 padding 会显著拖慢正常失败路径。
  请总工程师决策：是否接受固定周期 padding 带来的性能影响。

[0056] High    - 模型输出缺少对抗样本/恶意模式检测
  原因：当前 ActionValidator 已拦截常见 shell 元字符；更完整的对抗检测需要模型输出语义过滤。
  请总工程师决策：是否引入独立的输出内容安全分类器。

[0063] High    - 设备动作缺少 HMAC/完整性校验
  原因：同 [0008]，需与 provider 协商签名。

[0065] High    - HTTPS 未做证书固定
  原因：证书固定会阻塞 provider 证书轮换，维护成本高。
  请总工程师决策：是否采用 HPKP/公钥固定，或改走 mTLS/私有代理。

[0071] High    - 跨境数据传输未做地域/合规控制
  原因：需要 jurisdiction 映射 + 用户法律基础确认。
  请总工程师决策：是否增加 data_residency 配置与合规阈值拦截。

[0072] High    - Authorization 头明文传输 API Key
  原因：若 provider 不支持请求签名，则无法避免；建议结合 [0065] 证书固定。

[0074] High    - requirements.txt 无哈希校验
  原因：生成哈希需要联网 pip-compile/ Poetry，且需持续维护 lock 文件。
  请总工程师决策：是否迁移到 poetry/pdm/pip-tools 并启用 --require-hashes。

[0075] High    - Gradio 前端静态资源缺少 SRI
  原因：Gradio 框架自身限制，需升级或替换前端。

[0080] High    - UI 中 api_key 组件值仍通过网络传输（虽然已设为只读）
  原因：彻底移除需重构 UI 输入列表；本次已将其设为只读并忽略 UI 输入，实现 env-only。

[0081] High    - Gradio WebSocket 可能绕过 Basic Auth
  原因：Gradio 框架问题，建议前置反向代理统一鉴权。

[0083] High    - Basic Auth 明文传输
  原因：已在 launch 中增加 ssl_keyfile/ssl_certfile 环境变量支持；完整方案需部署反向代理 + TLS。

[0085] High    - UI 输出框可能泄露敏感信息
  原因：需要定义敏感信息分类并在输出前打码。

[0086] High    - 缺少安全审计日志（WHO/WHAT/WHEN）
  原因：需要独立审计 logger、不可变日志文件、IP/用户/设备/session 字段。

[0088] High    - API Key 权限未分离（分析/完成共用同一 key）
  原因：需要 provider 支持作用域 key 或应用层多 key 管理。

[0089] High    - 安全事件缺少告警（SIEM/webhook）
  原因：需要 SecurityEvent 类与告警管道。

[0090] High    - ADB 子进程输出无大小限制
  原因：subprocess.run(capture_output=True) 会缓冲全部输出；需改为流式读取并限制大小。

[0091] High    - ADB 远程截图文件在崩溃时残留
  原因：可在 __init__ 增加远程 prefix 清理，但会误删其他进程文件。

[0092] High    - 剪贴板回退导致敏感文本全局暴露
  原因：已增加用后清空，但更好的方案是 UiAutomator/输入法服务直接输入。

[0093] High    - am broadcast clipper.set 存在 Intent 注入面
  原因：建议替换为专用安全输入服务。

[0094] High    - 未校验设备序列号/指纹
  原因：需要 PHONEDRIVER_DEVICE_FINGERPRINT 配对流程。

[0095] High    - 设备动作缺少结构化审计日志
  原因：同 [0086]。

[0096] High    - adb 命令使用相对路径，PATH 可被篡改
  原因：已在代码中改为 shutil.which 解析绝对路径（若系统未找到 adb 则回退），并检查 world-writable。

[0097] High    - HOME 可被重定向以劫持 adb key
  原因：当前白名单保留 HOME；建议增加 adbkeys 目录权限校验。

[0098] High    - 未检测 ADB root 模式
  原因：可在运行前执行 adb shell id -u 检测 uid=0 并拒绝。

[0102] High    - Windows Junction Point 绕过 is_symlink()
  原因：需 Windows 专用 API（ctypes/GetFileAttributes）检测 reparse point。

[0105] High    - 未检测截图对抗样本
  原因：需要对抗鲁棒性模型或预处理，超出本次范围。

[0107] High    - cleanup_old 可能删除正在被读取的截图
  原因：已通过 _active_reads 集合并配合 provider 的 acquire_read/release_read 部分修复，但 provider 与 ScreenshotCapture 的耦合可进一步优化。

[0108] High    - 截图本地未加密
  原因：建议引入 OS keyring + AES-256-GCM 加密，但会显著改变存储/读取流程。

[0109]~[0115] Medium/High - PyPI 占位、SBOM、SLSA、build isolation、release signing、 consent banner 等
  原因：属于 DevSecOps/发布流程改造，需团队决策。
